End-to-end walkthrough: research → scaffold → flash → debug → reverse-engineer → successful wake.

## Hardware / software setup

- **ESP32**: ESP32-D0WDQ6 rev 1.0 (classic ESP32-WROOM-32, dual-mode BT, 2MB flash)
  - USB UART: CP2102, at `/dev/cu.usbserial-0001`
- **Target**: Nintendo Switch 2 with Joy-Cons attached, Pro Controller paired over classic BT
- **Toolchain**: PlatformIO (espressif32 platform, framework=espidf), NimBLE stack
- **Dev env**: macOS, VS Code + PlatformIO extension

## Step-by-step walkthrough

### 1. Convert ESP-IDF scaffold to PlatformIO

Initial scaffold used raw ESP-IDF (`CMakeLists.txt` + `main/`). Converted to PlatformIO layout:
- `platformio.ini` with `board=esp32dev`, `framework=espidf`
- `src/` instead of `main/`
- `sdkconfig.defaults` for NimBLE-only config

**Pitfall encountered**: setting `CONFIG_BT_NIMBLE_ROLE_CENTRAL=n` to minimize the build broke the link — `ble_hs_resolv.c` unconditionally references `ble_sm_alg_encrypt`, which only compiles when a connecting role is enabled. **Fix**: leave all roles at their Kconfig defaults (all `y`).

### 2. First target MAC attempt — WRONG

Assumed Nintendo BT MAC = Wi-Fi MAC + 1 byte offset. Flashed, tested — no wake.

### 3. 4-candidate cycling firmware

Cycled through four offsets (`+0, +1, +2, -1`) back-to-back with LED-blink identification. Auto-burst fired each candidate in sequence. **None worked.**

**Lesson**: Nintendo assigns Wi-Fi and BT MACs independently. The ESP32's shared-radio offset pattern is an Espressif implementation detail, not a universal convention.

### 4. LED signaling added

Added GPIO2 LED feedback (3 boot blinks, solid ON during burst, 5 fast blinks for error). Crucial for confirming the firmware actually ran without trusting just the serial log.

### 5. BLE scanner pivot

Swapped firmware to BLE scanner mode. First scan: 45 seconds, zero Nintendo-marked devices. Switch 2 itself doesn't advertise on BLE (it's a BLE central, not peripheral).

### 6. BLE scanner round 2 — ALL filters off, ACTIVE scan

- Relaxed filter: logged every device regardless of RSSI
- Switched to active scan (send scan_req to capture scan responses)
- **User protocol**: Entered Change Grip/Order on Switch 2, then detached/reattached both Joy-Cons during the capture

**Breakthrough**: captured Joy-Con advertisements during their reconnect window. Two Nintendo-marked devices — the left and right Joy-Cons. Pro Controller **NOT captured** (confirmed: it's classic BT only, invisible to BLE scanner).

### 7. Joy-Con advertisement decoded — Change Grip/Order state

Both Joy-Cons broadcast the SAME 31-byte payload structure, differing only by PID:

```
Offset  Bytes              Meaning
--------------------------------------------------------
  0     02 01 06           BLE Flags (LE General Discoverable)
  3     1B FF              Manufacturer data, 27 bytes follow
  5     53 05              Company ID 0x0553 (Nintendo)
  7     01 00 03           Header
 10     7E 05              Nintendo USB VID 0x057E
 12     XX 20              PID: 0x2066 or 0x2067 (Joy-Con L or R)
 14     00 01 00           Bytes 14-16 (byte 16 is a STATE FLAG)
 17     <6 bytes reversed> Target Switch 2 BT MAC (reversed)
 23     0F 00 00 00 00 00 00 00   Trailing pad
```

**Discovery: the Switch 2's BT MAC was embedded in bytes 17-22 of the Joy-Con's broadcast** — reversed. We had been searching for the Switch 2's MAC on-air; it turns out the Joy-Cons freely broadcast their paired host's MAC every time they disconnect.

### 8. Flashed wake firmware with captured target MAC

Built a wake firmware with:
- Target Switch 2 MAC extracted from Joy-Con broadcast
- Source MAC impersonating Joy-Con (via `esp_base_mac_addr_set`)
- Byte 16 = `0x00` (as captured)

**Result**: no wake.

### 9. Second scan — during actual Sleep mode

Re-ran the scanner while the Switch 2 was in **Sleep**, user pressed HOME on the detached right Joy-Con (the Switch woke from the real Joy-Con, confirming wake-from-Sleep works via a real controller).

**Key discovery**: payload now had byte 16 = `0x81`, not `0x00`.

**Interpretation**: byte 16 is a controller state flag.
- `0x00` = "open pairing mode" (Change Grip/Order state — what we'd captured first)
- `0x81` = "paired peer attempting reconnect to known host in Sleep state"

Only `0x81` is recognized by the Switch 2's wake-watch firmware as a legitimate trigger.

### 10. Updated firmware → WAKE SUCCESS

Flashed with byte 16 = `0x81` + captured Joy-Con source MAC + captured Switch 2 target MAC. Auto-burst fired. **Switch 2 woke from Sleep.**

## Final winning configuration

| Parameter | Source |
|---|---|
| Target Switch 2 BT MAC | Captured from Joy-Con broadcast (bytes 17-22 reversed) |
| Source advertiser MAC | Captured Joy-Con BT MAC |
| PID | From Joy-Con's own broadcast (0x2066 or 0x2067) |
| **Byte 16** | **`0x81`** — the critical state flag |
